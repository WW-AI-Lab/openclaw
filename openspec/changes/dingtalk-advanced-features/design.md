## Context

钉钉支持互动卡片（Interactive Card），这是实现流式输出的关键技术。互动卡片可以在发送后被更新，实现"打字机效果"——AI 每生成一段内容就更新卡片，用户实时看到生成进度。

飞书插件的流式输出实现（`streaming-card.ts`）是成熟的参考：

1. 创建一张空卡片发送给用户
2. AI 每产生一段输出就通过 API 更新卡片内容
3. 生成完成后发送最终版本

钉钉的互动卡片 API：

- 创建卡片实例：`POST /v1.0/card/instances`
- 更新卡片内容：`PUT /v1.0/card/instances`
- 卡片模板支持 Markdown 内容

媒体方面，钉钉消息中包含 `downloadCode` 或 `pictureDownloadCode`，可通过 API 下载文件内容。上传则通过上传文件接口获取 mediaId 后在消息中引用。

## Goals / Non-Goals

**Goals:**

- 实现流式卡片输出，在 AI 生成过程中实时更新显示
- 实现媒体文件下载（从接收的图片/文件/音频/视频消息中提取文件）
- 实现媒体文件发送（上传文件并通过消息发送）
- 编写完整的用户文档
- 实现群组级别精细化配置
- 实现 typing 指示器

**Non-Goals:**

- 互动卡片的按钮回调处理（可作为未来增强）
- 钉钉文档/表格/知识库等工具集成（可作为独立提案）
- 视频/语音通话集成

## Decisions

### D1: 流式卡片实现方式

**选择：使用钉钉互动卡片 API + Markdown 模板**

实现流程：

1. AI 开始生成 → 创建空白互动卡片 → 发送给用户
2. AI 每产生 ~100 字符 → 更新卡片 Markdown 内容
3. AI 完成生成 → 更新卡片为最终内容 + 设置 "streaming: false" 标记
4. 更新频率限制：最快 500ms 一次，避免 API 限流

备选方案：使用多条消息模拟流式。放弃原因：消息刷屏体验差。

### D2: 媒体下载策略

**选择：使用 downloadCode API 按需下载**

钉钉提供两种下载方式：

1. `downloadCode`：通过 `/v1.0/robot/messageFiles/download` 下载（推荐）
2. `fileId`：通过企业存储 API 下载

选择 `downloadCode` 方式，因为它直接对应消息中的文件，不需要额外的存储权限。

### D3: 文档结构

参考飞书文档（`docs/channels/feishu.md`）的结构：

1. 概述和状态
2. 快速开始（安装插件 → 创建应用 → 配置凭证）
3. 详细配置（访问控制、群组配置、多账号等）
4. 支持的消息类型
5. 故障排除
6. 配置参考

### D4: 群组精细化配置

**选择：与飞书保持一致的 groups 配置结构**

```jsonc
{
  "channels": {
    "dingtalk": {
      "groups": {
        "cidXXXX==": {
          "requireMention": false, // 该群组无需 @
          "enabled": true,
        },
      },
    },
  },
}
```

## Risks / Trade-offs

### R1: 互动卡片 API 的限制

**风险**：钉钉互动卡片 API 可能有调用频率限制。

**缓解**：

- 设置 500ms 的最小更新间隔
- 合并短时间内的多次更新
- 如果更新失败，降级为普通文本消息

### R2: 媒体文件大小限制

**风险**：钉钉可能对文件上传/下载有大小限制。

**缓解**：

- 设置 `mediaMaxMb` 配置项，默认 30MB
- 超过限制时提示用户文件过大

### R3: 互动卡片模板兼容性

**风险**：钉钉互动卡片的 Markdown 支持可能与标准 Markdown 有差异。

**缓解**：

- 测试卡片中 Markdown 的渲染效果
- 对不支持的 Markdown 语法进行转换或降级
